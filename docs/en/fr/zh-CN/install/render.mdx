Déployez OpenClaw sur Render en utilisant l'infrastructure en tant que code. Le Blueprint `render.yaml` intégré définit de manière déclarative votre pile technologique complète — services, disques, variables d'environnement — vous permettant de déployer en un seul clic et de gérer votre infrastructure en même temps que votre code.

## Prérequis

- Un [compte Render](https://render.com) (offre un plan gratuit)
- Une clé API de votre [fournisseur de modèle](/providers) préféré

## Déployer avec le Blueprint Render

<a
  href="https://render.com/deploy?repo=https://github.com/openclaw/openclaw"
  target="_blank"
  rel="noreferrer"
>
  Déployer sur Render
</a>

Cliquer sur ce lien va :

1. Créer un nouveau service Render basé sur le Blueprint `render.yaml` à la racine du dépôt.
2. Vous inviter à configurer `SETUP_PASSWORD`
3. Construire l'image Docker et déployer

Une fois le déploiement terminé, l'URL de votre service sera au format `https://<service-name>.onrender.com`.

## Comprendre le Blueprint

Un Blueprint Render est un fichier YAML qui définit votre infrastructure. Le `render.yaml` dans ce dépôt configure tout ce qui est nécessaire pour exécuter OpenClaw :

```yaml
services:
  - type: web
    name: openclaw
    runtime: docker
    plan: starter
    healthCheckPath: /health
    envVars:
      - key: PORT
        value: "8080"
      - key: SETUP_PASSWORD
        sync: false # prompts during deploy
      - key: OPENCLAW_STATE_DIR
        value: /data/.openclaw
      - key: OPENCLAW_WORKSPACE_DIR
        value: /data/workspace
      - key: OPENCLAW_GATEWAY_TOKEN
        generateValue: true # auto-generates a secure token
    disk:
      name: openclaw-data
      mountPath: /data
      sizeGB: 1
```

Fonctionnalités clés du Blueprint utilisées :

| Fonctionnalité        | Objectif                                           |
| --------------------- | -------------------------------------------------- |
| `runtime: docker`     | Construire à partir du Dockerfile du dépôt        |
| `healthCheckPath`     | Render surveille `/health` et redémarre les instances défaillantes |
| `sync: false`         | Demander la valeur lors du déploiement (pour les secrets) |
| `generateValue: true` | Générer automatiquement une valeur sécurisée      |
| `disk`                | Stockage persistant, les données survivent aux redéploiements |

## Choisir un plan

| Plan      | Mécanisme de mise en veille | Disque | Cas d'usage              |
| --------- | --------------------------- | ------ | ------------------------ |
| Free      | Mise en veille après 15 min d'inactivité | Non disponible | Tests, démos |
| Starter   | Jamais                      | 1GB+   | Usage personnel, petites équipes |
| Standard+ | Jamais                      | 1GB+   | Production, multi-canal  |

Le Blueprint utilise `starter` par défaut. Pour utiliser le plan gratuit, définissez `plan: free` dans votre `render.yaml` forké (notez que l'absence de disque persistant signifie que la configuration sera réinitialisée après chaque déploiement).

## Après le déploiement

### Terminer l'assistant de configuration

1. Visitez `https://<your-service>.onrender.com/setup`
2. Entrez votre `SETUP_PASSWORD`
3. Sélectionnez votre fournisseur de modèle et collez votre clé API
4. Configurez optionnellement les canaux de messages (Telegram, Discord, Slack)
5. Cliquez sur **Run setup**

### Accéder au tableau de bord

Le panneau d'administration web se trouve à `https://<your-service>.onrender.com/openclaw`.

## Fonctionnalités du tableau de bord Render

### Journaux

Consultez les journaux en temps réel dans **Dashboard → Votre service → Logs**. Filtrez par type :

- Journaux de construction (création d'image Docker)
- Journaux de déploiement (démarrage du service)
- Journaux d'exécution (sortie de l'application)

### Accès Shell

Pour déboguer, ouvrez une session shell via **Dashboard → Votre service → Shell**. Le disque persistant est monté sur `/data`.

### Variables d'environnement

Modifiez les variables dans **Dashboard → Votre service → Environment**. Les modifications déclenchent un redéploiement automatique.

### Déploiement automatique

Si vous utilisez le dépôt OpenClaw original, Render ne déploiera pas automatiquement votre OpenClaw. Pour le mettre à jour, exécutez manuellement la synchronisation du Blueprint dans le tableau de bord.

## Domaines personnalisés

1. Allez à **Dashboard → Votre service → Settings → Custom Domains**
2. Ajoutez votre domaine
3. Suivez les instructions pour configurer le DNS (CNAME pointant vers `*.onrender.com`)
4. Render configurera automatiquement le certificat TLS

## Mise à l'échelle

Render supporte la mise à l'échelle horizontale et verticale :

- **Mise à l'échelle verticale** : Changez de plan pour plus de CPU/mémoire
- **Mise à l'échelle horizontale** : Augmentez le nombre d'instances (plans Standard et supérieurs)

Pour OpenClaw, la mise à l'échelle verticale est généralement suffisante. La mise à l'échelle horizontale nécessite des sessions persistantes ou une gestion d'état externe.

## Sauvegarde et migration

Exportez votre configuration et votre espace de travail à tout moment :

```
https://<your-service>.onrender.com/setup/export
```

Cela téléchargera un fichier de sauvegarde portable que vous pouvez restaurer sur n'importe quel hôte OpenClaw.

## Dépannage

### Le service ne démarre pas

Vérifiez les journaux de déploiement dans le tableau de bord Render. Problèmes courants :

- `SETUP_PASSWORD` manquant — Le Blueprint vous le demandera, mais confirmez qu'il est défini
- Port non correspondant — Assurez-vous que `PORT=8080` correspond au port exposé par le Dockerfile

### Démarrage à froid lent (plan gratuit)

Les services du plan gratuit se mettent en veille après 15 minutes d'inactivité. La première requête après la mise en veille prend quelques secondes pour que le conteneur démarre. Passez au plan Starter pour un service toujours en ligne.

### Perte de données après redéploiement

Cela se produit sur le plan gratuit (pas de disque persistant). Passez à un plan payant ou exportez régulièrement votre configuration via `/setup/export`.

### Échec du contrôle de santé

Render s'attend à une réponse 200 de `/health` dans les 30 secondes. Si la construction réussit mais le déploiement échoue, le service peut prendre trop de temps à démarrer. Vérifiez :

- Les erreurs dans les journaux de construction
- Si le conteneur s'exécute correctement localement avec `docker build && docker run`
